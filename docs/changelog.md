# Changelog

## 2026-07-14

- Drafted project spec (`docs/project-spec.md`) via `/grill-me` interview: architecture, tech stack, database schema, and open dependencies finalized.
- Initialized git repo, added `.gitignore`, `.env.example`, project `CLAUDE.md`.
- Scaffolded Node/TypeScript project (`package.json`, `tsconfig.json`).
- Built and applied the `scan_results` Postgres migration to Neon (`src/db/schema.sql`, `src/db/migrate.ts`).
- Built the scanner module (`src/scan/pagespeed.ts`, `src/scan/index.ts`) - calls PSI for each URL x device with basic concurrency limiting, per-URL error isolation, and writes results to Neon.
- Verified end-to-end against the real homepage (https://www.menkind.co.uk/): fixed two bugs found during verification - non-integer Lighthouse ms values needed rounding before insert, and Lighthouse 13 renamed `render-blocking-resources`/`third-party-summary` to `render-blocking-insight`/`third-parties-insight` (parser now checks both). Also fixed console-error/deprecation "source" extraction, which was reading a category label instead of the actual source URL.
- Built the flagging module (`src/flag/`): Google's official CWV rating bands for LCP/CLS, Lighthouse's own bands for performance score, and a documented TBT-as-INP-lab-proxy band kept separate from the real INP thresholds. Regression detection compares each run against a trailing 7-run average, flagging swings of 20%+ once at least 3 prior runs exist. Verified both the threshold and regression paths against real and synthetic data.
- Fixed two `node-postgres` type-parsing bugs found while building the report generator: `DATE` columns were coming back as JS `Date` objects at local midnight, and `.toISOString()` silently shifted the date backward a day under the UK's non-zero UTC offset; and `NUMERIC` columns (`cls`) were returned as strings, breaking `.toFixed()`. Both fixed globally via custom `pg` type parsers in `src/db/client.ts` rather than patched per-callsite.
- Built the report generator (`src/report/`): parameterized by date range (`npm run report` for today, `report:weekly`/`report:monthly` for rollups, `report -- range <start> <end>` for anything else), reuses the flagging module, and renders Markdown following the report-content rules from the spec (specifics for every count, per-page attribution, fact vs. inference, rationale-ordered recommendations, glossary, references). Verified against real data for all four modes.
- Built the dashboard builder (`src/dashboard/`): static `site/index.html` with summary cards, inline-SVG LCP/perf-score trend sparklines (no charting library dependency), and a current-snapshot table; converts generated Markdown reports to HTML via `marked`. Solved the "Pages deployment replaces everything" problem by having the dashboard link (not regenerate) previously-published reports via an optional `previousReportsDir`. Verified visually in-browser, not just typechecked - confirmed content, console had no errors, and a report page rendered correctly.
- Built the four GitHub Actions workflows (`daily-scan`, `weekly-rollup`, `monthly-rollup`, `custom-range-report`), publishing via `peaceiris/actions-gh-pages` with `keep_files: true` so historical reports survive each redeploy. YAML validated with PyYAML but not yet run on a real GitHub Actions runner. GitHub Pages still needs a one-time manual enable (branch-based source) once the `gh-pages` branch exists after the first run.
- Slack notifier deliberately left for last per Samuel's instruction - everything else in the pipeline is otherwise complete.
- Samuel reviewed the live dashboard and flagged two gaps: the UI needed real design work, and whether real-user network conditions were considered at all. They weren't - the scanner only ever captured Lighthouse's simulated lab run, never PSI's `loadingExperience`/`originLoadingExperience` fields (real Chrome UX Report data), despite that being part of the original architecture pitch. Checked a live PSI response for the homepage: lab and field data actively disagreed (lab rated LCP "poor" at 9827ms mobile, field rated it "good" at 2372ms p75; lab rated CLS "good" at 0.013, field rated it "poor" at 0.26).
- Added CrUX field data capture (`src/scan/pagespeed.ts`): page-level data preferred, falls back to site-wide origin data for lower-traffic pages, `field_data_source` records which. New `rateInp` threshold function uses the official INP bands (distinct from the TBT-proxy bands) now that real INP is available. Migration is additive (`alter table ... add column if not exists`) so it re-runs safely against the existing table.
- Refactored the duplicated `scan_results` SELECT column list (previously drifted between `flag/index.ts` and `report/queries.ts`, which is exactly how the new field columns would have silently gone missing from flagging) into a single `src/db/queries.ts` both modules import.
- Updated the flagging module to prefer field data over lab data per-metric when available, with messages that say clearly which data source and thresholds were used. Updated the report generator with a new "Lab Data vs Real-User Field Data" section that shows both and explicitly calls out disagreements rather than silently resolving them.
- Rebuilt the dashboard against the `dataviz` skill: read the skill's form/color/marks/interaction references, validated the categorical and status palettes with `scripts/validate_palette.js` (all checks passing in both light and dark) before using them, replaced the two separate mobile/desktop sparklines-per-metric with one two-series line chart per metric (per the skill's "one axis" rule - same unit, same chart, legend distinguishes the series), added a hover crosshair+tooltip built in vanilla JS/SVG (no charting library), and status badges that always pair color with an icon and text label (never color alone).
- Caught and fixed a real XSS-pattern bug during the chart build: the tooltip was assembling `innerHTML` via string concatenation, which the dataviz skill's own interaction guidance explicitly warns against. Rebuilt it with `createElement`/`textContent`.
- Verified the redesign in-browser via computed-style and simulated-interaction checks rather than a screenshot - the Browser pane's screenshot/zoom capture was timing out at the tool level this session (unrelated to the page: console was clean, network requests all 200'd, and the DOM read back structurally correct). Computed colors matched the validated palette exactly (e.g. mobile series `#2a78d6` light / `#3987e5` dark) in both color schemes, and a simulated hover correctly moved the crosshair and populated the tooltip with the right date/values. Caught and fixed one real bug this way: the legend swatch was hardcoded to the dark-mode color regardless of active scheme, so it mismatched the line color in light mode.
